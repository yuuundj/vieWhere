# 📚 vieWhere - 기반 지식 및 기술 스택 가이드

이 문서는 본 프로젝트를 성공적으로 구현하기 위해 필수적으로 사용되는 프로그래밍 언어, 프레임워크, 그리고 기술적 핵심 개념에 대한 사전 지식을 정리한 가이드입니다. 

---

## 1. 프레임워크 & 툴체인 (Framework & Toolchain)

### ⚛️ React 18+ (UI 라이브러리)
- **개념**: 사용자 인터페이스(UI)를 구축하기 위한 Component(부품) 기반의 JavaScript 라이브러리입니다.
- **필요한 이유**: 야구장에는 수많은 구역이 존재하며, 사용자의 클릭에 따라 화면(시야 사진, 구역 정보 등)이 동적(Reactive)으로 변해야 합니다. React는 `State` 관리를 통해 이러한 상호작용을 가장 빠르고 효과적으로 처리할 수 있게 해줍니다.

### ⚡ Vite (빌드 툴)
- **개념**: 빠르고 가벼운 차세대 프론트엔드 환경 구축 도구입니다.
- **필요한 이유**: 초기 세팅과 실행 속도가 무겁고 느린 Webpack 같은 기존 도구를 대체합니다. 코드를 수정하면 즉각적으로 브라우저에 변경 사항이 반영되어 압도적인 개발 편의성을 제공합니다.

### 🔷 TypeScript (언어)
- **개념**: JavaScript에 '정적 타입(Type) 검사' 기능을 더해 안정성을 높인 확장 언어입니다.
- **필요한 이유**: 구역별 데이터, 가격 정보, 이미지 주소 등 다양한 데이터가 오가는 이 앱에서 '데이터의 형태'를 미리 정의하여 에러를 사전에 막아주는 든든한 방어막 역할을 합니다.

---

## 2. 형상 및 코드 관리 (Version Control)

### 🐙 Git & GitHub
- **개념**: 파일의 변경 이력을 추적하는 분산 버전 관리 시스템(Git)과 이를 온라인으로 공유/저장하는 호스팅 서비스(GitHub)입니다.
- **필요한 이유**: 가상 야구장 테스트 구성을 완료한 다음, 본격적으로 수백 개의 SSG 랜더스필드 구역 데이터를 입력하는 등 복잡한 작업(Phase 3)을 진행하게 됩니다. 
이 단계에서 코드가 고장 나거나 예기치 못한 에러가 발생했을 때, 언제든 **'안전하게 동작하던 가상 테스트 완료 시점(Phase 2)'으로 코드를 되돌리기 위해(Rollback)** 필수적으로 사용됩니다.

## 3. 모형 및 디자인 구현 (Markup & Styling)

### 🎨 SVG (Scalable Vector Graphics)
- **개념**: 수학적 계산으로 픽셀이 아닌 선과 면을 그리는 '벡터' 방식의 2D 이미지 기술입니다.
- **필요한 이유**: **본 프로젝트의 가장 핵심 기술**입니다. 야구장 도면(맵)은 아무리 확대/축소해도 화질이 깨지지 않아야 합니다. 또한, 일반 그림 파일(.jpg, .png)과 달리 SVG 내부의 각 구역(`<path>`, `<polygon>` 등)에 프로그래밍 코드를 입혀 색깔을 바꾸거나 '클릭 이벤트'를 부여할 수 있기 때문에 채택되었습니다.

### 💅 Vanilla CSS (CSS Modules)
- **개념**: Tailwind 등 외부 기술을 빌리지 않는 순수한 오리지널 스타일링 기술입니다. 모듈화를 통해 컴포넌트끼리 디자인 코드가 충돌하는 것을 방지합니다.
- **필요한 이유**: 프리미엄 앱처럼 섬세하게 조절된 반응형 디자인, 유리 같은 질감(Glassmorphism), 구역을 눌렀을 때 자연스럽게 떠오르는 미세 애니메이션(Micro-interaction)을 가장 자유롭고 정밀하게 제어할 수 있습니다.

## 4. 필수 핵심 개념 (Core Concepts)

### 🔄 상태 관리 (State Management)
- 사용자가 현재 **어느 구장의 어느 블록을 선택하고 있는지** 프로그램이 지속적으로 기억하게 하는 기술입니다. 이 상태 값이 "1루 101블록"에서 "3루 202블록"으로 변하면, 우측의 사진 패널이 연쇄적으로 업데이트됩니다.

### 📱 반응형 프론트엔드 (Responsive & Mobile-First)
- 데스크톱 모니터뿐만 아니라, 실제 야구장 관람객들이 **이동 중 스마트폰에서 자주 열어볼 것**을 최우선으로 가정해야 합니다.
- 기기의 화면 너비에 따라 좌석 지도는 상단/좌측에, 사진 정보는 하단/우측으로 유동적으로 이동하게 만드는 레이아웃 설계 기법입니다.

### 🧱 JSON 데이터베이스 연동 구조
- 개별 구역에 대한 고유 식별자(ID)를 바탕으로 데이터 창고 역할을 하는 JSON 파일과 통신합니다.
- (예: 사용자가 SVG의 `id="ssg-friendly-zone"` 구역 클릭 ➡️ JSON에서 이 값을 검색 ➡️ 해당 구역의 `이미지 URL`, `티켓값` 텍스트를 UI에 전달하여 렌더링)
